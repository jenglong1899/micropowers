# micropowers

本项目**并非**为了快速产出代码而设计，而是为了代码质量而设计，现实一次次教育我 AI 写的代码必须被认真审查。本项目的指令能够让你以一种舒服的方式去审查代码。本项目还包含了一个比 codex 自带的更好的记忆指令。

你需要用到的文档有三个（文档都很简短，强烈建议去读一下）：

- `coding-like-a-human`: 人类写代码的时候可不会写一个200多行的计划文件，人类会用简短篇幅描述大致的结构，然后在代码中实现骨架（要有哪些类、接口），最后去实现这些骨架。

- `agentsmd-to-be-copied.md`: 选择你喜欢的那些指令，复制到 ~/.codex/AGENTS.md 中。**文档末尾有个记忆相关的指令，需要配合`memory-instruction.md`使用**

- `memory-instruction.md`: 把这个复制到 ~/.codex 目录下。

codex有记忆机制，为什么用我这个？我看过~/.codex目录里面的记忆，我对其不太满意，它的总体记忆居然有其他项目的记忆，我在搞项目A，为啥要把项目B的记忆也一块搞进去？

为啥不把这个放进 ~/.codex/AGENTS.md 里面？Agent 并不需要在工作过程中就开始做摘要，完全可以等到要准备重置之前再摘要。另外，在工作的过程中做摘要会分散 AI 的注意力。

那可不可以做成 stop hook 让它在完成工作后自动去阅读对应的指令？不能，因为 `coding-like-a-human` 的指令让它一次只写100行左右的代码，这也算一个stop

建议先在项目目录下的 AGENTS.md 中先亲自写一下大致架构是怎样的，先不要偷懒，剩下的再让 AI 去补全。

然后在其开头加上：
```本文档是对代码库的一种摘要，目的是为了帮助 Agent 建立对代码库的理解，而不是作为代码文件的替代，所以 Agent 不能光靠本文档就去回答用户的问题、制定计划，如果 Agent 违反了这个规则，会被惩罚扣分。```

## coding-like-a-human 的演进过程

如果你想知道这个 skill “为什么这么写而不那么写”，就需要了解它的演进过程。

最早期我写了一个 `plan-coding`，一开始表现得很好，但不知道怎么回事，26年2月份后 GPT 按照这个 skill 写出来的计划就很糟糕（该详细写的地方没详细写、不该详细写的地方又写了很多）。

于是我新写了一个 `simple-plan-coding`，先让AI写个简单的计划（基本一分钟就能看完），然后告诉它一次只写100行以内的代码，我在旁边盯着，一边写一边纠正（反正这些代码你迟早都是要读的）。

然后我意识到一个简单的事实：在古法编程时代，我们并不会严格地区分计划和写代码，整个编程过程更像是一个逐步细分的过程，我们先用简单的语言、图表描述我们要做什么，通常二三十行就可以搞定，肯定不会有一个100行、200行的计划文件，然后我们就开始搭建骨架（那二三十行计划的再一步细分），然后实现这些骨架（进一步细分）。

`plan-coding`的一个问题是，对于你要做的一个功能，他要做的改动可能是很大的，如果 AI 能完美地遵守里面的那条指令，它给出的计划文件会很长，可能超过200行，读这么长的文件是反人性的，读了这么久都没有一行代码产出，你会很抓狂。既然这些计划文件描述的都是大概的骨架，那为什么不一开始就用代码把骨架表达出来呢？
