# Plan and Code

把“做计划”和“写代码”视为同一个逐步细化过程。目标不是先产出一份完整计划，再一次性实施，而是让用户用最低的理解成本看清方向、结构和行为，并能在每一轮及时纠偏。

```text
目标与约束
    ↓
候选方向
    ↓ 用户确认
结构 / 数据流 / 测试场景
    ↓
代码骨架
    ↓
局部实现
    ↓
可验证的完整实现
```

每一轮只细化一层。根据内容选择最容易理解的表达载体：能用少量文字讲清楚就用文字；关系复杂时用 ASCII 图；类、接口、数据结构或调用关系用代码更直观时，直接修改代码来表达。

## 工作原则

### 以理解成本为准，而不是以“是否属于计划”为准

不要为了维持传统计划形式，把本可由代码清楚表达的内容改写成几十行文字。

例如，有时候与其逐项描述“新增哪些类、每个类有哪些方法、参数如何传递”，不如直接写出类、接口、函数签名和关键类型。与其用文字罗列测试场景，不如先写测试名称、输入和预期结果。

反过来，不要为了显得有进展而写大量空壳代码。若一张小图或一段话更清楚，就不要创建文件和抽象。

### 代码也是设计稿

允许在方案阶段新增、修改或删除工作代码。代码骨架可以暂时不完整，也不要求每一轮结束时项目都能运行，因为半完成的业务改动即使能编译，其行为也没有任何价值。

但是，要明确告诉用户当前代码处于哪种状态：

- 仅用于表达结构，暂时不能运行；
- 主路径已实现，边界情况尚未处理；
- 已完整实现，等待验证。

不要用无意义的占位实现伪造“可运行”，例如返回错误的默认值、吞掉异常或保留已经确定不需要的旧路径。若旧代码会干扰新结构的理解，可以直接删除；不要仅为保持中间状态可运行而同时维护新旧两套逻辑。

### 每轮都要形成一个容易判断的问题

每一轮的产物都应帮助用户回答一个具体问题，例如：

- 这个技术方向是否正确？
- 职责和边界是否合理？
- 接口是否符合使用习惯？
- 关键业务行为是否正确？

不要一次展开所有细节。用户尚未确认上层方向时，不要提前实现依赖该方向的大量细节。

## 工作流程

### 1. 校准目标

先检查用户要求是否值得做、是否存在明显风险，以及真正目标是否有更简单的实现方式。能合理推断的内容作为假设继续推进，并明确写出会影响方案的假设。

如果缺失信息会导致完全不同的技术路线、不可逆的数据变更或无法判断正确性，停下来询问用户。

### 2. 给出最小候选方向

用一小段话、一个小图或少量伪代码说明一个或多个候选方向，只写足以支持选择的差异、收益和代价。推荐一个方向并说明理由，然后等待用户选择。

不要在这一阶段写完整实施清单。

### 3. 把选定方向细化一层

只展开下一层最重要的实现结构和测试思路。优先选择最直观的产物：

```text
内容                         优先表达方式
────────────────────────────────────────────
职责、依赖、状态转换         ASCII 图或短文
模块、类、接口、函数签名     代码骨架
数据形状、字段和约束         类型、模型或 schema
请求与处理流程               路由/控制器骨架 + 流程图
业务行为和边界条件           测试名称、样例或测试骨架
简单的局部逻辑               直接实现
```

可以直接修改现有代码，也可以新建文件。只写当前决策所需的结构，不提前加入“以后可能有用”的扩展点。

### 4. 展示本轮结果并等待判断

简短说明：

1. 本轮确定了什么；
2. 哪些文件或结构发生了变化；
3. 当前代码能否运行、哪些部分仍是骨架；
4. 下一轮准备细化什么；
5. 当前需要用户判断的唯一或少量关键问题。

不要再额外生成一份与代码重复的长计划。若需要计划文件，只记录代码无法自然承载的决策、假设、图表和未决问题，并随着设计变化同步更新。

### 5. 继续逐层实现

获得用户确认后，再把骨架细化为局部实现、把测试场景细化为可运行测试，并逐步处理错误路径和边界条件。重复“细化一层 → 展示 → 确认”，直到实现完整且验证通过。

## 控制每轮范围

- 默认让一轮改动保持在用户几分钟内可以读懂的规模；不要机械追求固定行数。
- 若改动虽短但包含多个关键决策，拆成多轮。
- 若代码是重复、机械且没有新的设计决策，可以一次完成，不必按行数强行切割。
- 不要把格式化、生成文件或删除废弃代码算成需要用户逐段确认的业务设计。
- 用户明确要求连续实施时，可以连续推进已确认方向；遇到新的关键决策再停下来。

## 验证与提交

代码仍是结构草稿时，不要为了通过测试而添加错误的临时逻辑；可以运行静态检查或针对已完成局部的测试，并如实报告未通过的原因。

进入可验证实现后，运行与改动范围匹配的测试。调试时通过日志或实际运行结果定位问题，不要只靠阅读代码猜测。

只有用户明确要求提交，或双方事先约定每轮提交时，才创建 Git commit。提交前检查工作区，避免把用户的无关改动包含进去。

## 完成标准

直到以下条件都满足，才把工作视为完成：

- 用户确认的行为已经实现，而不只是存在代码骨架；
- 已移除被新方案取代且不再需要的旧代码；
- 关键正常路径、错误路径和边界条件已验证；
- 代码附近的文档、注释和 ASCII 图仍与实现一致；
- 最终说明足够简短，用户无需再阅读一份重复代码内容的长计划。
