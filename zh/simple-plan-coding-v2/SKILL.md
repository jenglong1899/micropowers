## 步骤

以下提到的“思路”，你也可以理解为“计划”、“方案”，它们的意思是相似的。

整个计划过程就是把想法一步一步细化的过程。

计划的一个要求是：让用户以最低难度来理解计划

1. 思考用户提出的方案有没有什么雷点，有没有更好的方案？（没有什么是比高效地做一件本不该做的事更低效的了）

2. 给出一个或多个初步思路，每个思路用一小段话来描述，问用户同意哪个初步思路。

3. 对用户同意的初步思路，给出更细分一层的(1)实现思路和(2)测试思路（注意这里说的是细分一层，不是让你做非常细分的），尽量用50行以内的篇幅描述大概的思路（文字、图表均可），新建一个文件来存储这个思路

对比一下两种方式：一种是用文字、图表去描述做哪些改动，另外一种方式是直接去改代码【比如写代码的骨架（有哪些类，有哪些接口）】。不要有“每次做完改动后代码都要能跑”的想法，比方说我们要修改某个功能，那就涉及删除已有的一些代码，添加一些代码，这里追求这个代码要能跑完全没有任何意义，因为你这个修改都没有修改完全，其业务逻辑完全是错的，没有任何意义。

哪种方式能让用户以最低难度来理解，那就去做哪种。

5. 继续细分，比如开始实现一些类，实现一些接口。反复这个过程


## 注意事项
- 遇到关键问题就停下来问，能合理猜测的就继续工作，并在计划中明确写出该假设。“关键问题”包括但不限于：导致计划不可执行或技术路线分歧很大的问题；无法作出合理假设的问题。
- 在用户明确表示开始实施之前，我们始终处于计划制定阶段，期间不要修改任何已有的代码文件。
- 在代码的过程中，如果设计有变动，也要修改原计划文件
- 若现有代码存在影响工作的缺陷（例如文件过大、边界模糊、职责混乱），应在设计方案中包含针对性改进
- 一次只能写100行以内的代码，然后等待用户确认，用户确认后 commit 一次备份一下，commit message用"partial-feat/refactor/fix:..."开头，然后继续。commit之前用git status检查一下就行，不要用 git diff。
