## 步骤

以下提到的“思路”，你也可以理解为“计划”、“方案”，它们的意思是相似的。

1. 思考用户提出的方案有没有什么雷点，有没有更好的方案？（没有什么是比高效地做一件本不该做的事更低效的了）

2. 给出一个或多个初步思路，每个思路用一小段话来描述，问用户同意哪个初步思路。

3. 对用户同意的思路，给出更细分一层的(1)实现思路和(2)测试思路（注意这里说的是细分一层，不是让你做非常细分具体的）。新建一个文件来存储这个思路
  - **要求1：自顶向下写思路：为了实现用户的需求需要有怎样的抽象 -> 抽象的大概实现方式**
  - **要求2：计划请务必保持精简，其行数尽量不要超过“开始实施后，代码的变化的行数”的1/10**。能用50行写清楚，就不要用100行。同样的一个意思在计划文档的前面写过了，计划文档后面就不要再重复了。
  - 只要能实现这2个要求，你可以自由发挥，比如用ASCII/Mermaid图表、伪代码、自然语言写的思路等等。

4. 当用户同意开始实施后，开始工作，一次只能写100行以内的代码，然后等待用户确认，用户确认后 commit 一次备份一下，commit message用"partial-feat/refactor/fix:..."开头，然后继续。commit之前用git status检查一下就行，不要用 git diff。

## 注意事项
- 遇到关键问题就停下来问，能合理猜测的就继续工作，并在计划中明确写出该假设。“关键问题”包括但不限于：导致计划不可执行或技术路线分歧很大的问题；无法作出合理假设的问题。
- 在用户明确表示开始实施之前，我们始终处于计划制定阶段，期间不要修改任何已有的代码文件。
- 在代码的过程中，如果设计有变动，也要修改原计划文件
- 若现有代码存在影响工作的缺陷（例如文件过大、边界模糊、职责混乱），应在设计方案中包含针对性改进
