# Plan and Code

**目标：让用户用最低的理解成本确认方案。**

本文档描述的规则都是启发式规则，并不是强制要求，也没有涵盖你所有应该做或者不应该做的事情，只要能实现这个目标，你可以自由发挥。

把工作看成连续细化，而不是“先写完计划，再开始编码”：

```text
目标与约束 -> 初步方案 -> 图表/文字/代码骨架 -> 局部实现 -> 完整实现
```

每一轮只细化一层。选择最容易理解的表达方式：能用简短文字、图表讲清楚就写图文；用代码骨架更直观时，就直接修改代码，不要用一两百行文字预先描述几十行（骨架）代码能够更准确表达的设计。

## 工作方式

1. 检查用户提出的目标和方案，指出风险、不必要的工作及更简单的路线（没有什么是比高效地做一件本不该做的事情更低效的了）
2. 用短段落给出一个或几个初步方案，让用户确认方向；方向确认前只调查和讨论，不修改代码。
3. 新建一份过程文档（尽量不超过50行），只记录代码不方便表达的东西，比如假设和待确认问题。文档格式不强制要求是 MD 文档，如果用 HTML 能让用户更容易理解的话，就用 HTML。
4. 用户确认方向后，进入渐进式实现。可以删除确定不再需要的代码，也可以先加入模块、类、接口、函数签名和数据结构等骨架；此时骨架就是方案的一部分，不必再用长篇文字重复描述。
5. 向骨架逐步填入实现和测试。每轮只让用户判断一个层次或一个关键问题，通常要把业务代码改动控制在 50~70 行左右，这里不是强制要求，不要为了遵守行数限制去拆开一个完整、容易理解的结构。
6. 每轮结束后停下来，等待用户去检查改动，等用户确认后，创建检查点 commit，然后继续。

## 注意事项

- 遇到会改变技术路线或使方案不可执行的关键问题时，停下来询问；能够合理推断的内容继续推进，并明确记录假设。
- 现有代码若存在直接妨碍本任务的边界或职责问题，把必要的整理纳入方案
- 在实现过程中，不要有“每次做完改动后代码都要能跑”的想法，比方说我们要修改某个功能，那就涉及删除已有的一些代码，添加一些代码，这里追求这个代码要能跑完全没有任何意义，因为你这个修改都没有修改完全，其业务逻辑完全是错的，没有任何意义。
